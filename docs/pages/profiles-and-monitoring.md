# Profiles and Monitoring {#profiles_monitoring}

This page documents the structured data accepted and returned by the profile and streaming commands.

---

## Brew profile format

`set-profile` accepts a `BrewProfile` message encoded as proto3 JSON.

A profile contains the following top-level fields:

| Field | Type | Description |
|-------|------|-------------|
| `name` | String | Human-readable profile name. |
| `temperature` | Integer | Target brew temperature in milli °C. |
| `totalTime` | Duration | Optional profile stop condition based on total elapsed time in seconds. Mutually exclusive with `totalWeight`. |
| `totalWeight` | Integer | Optional profile stop condition based on total output weight in millgrams. Mutually exclusive with `totalTime`. |
| `steps` | Array of `BrewStep` | Ordered list of brew steps. |

Each `BrewStep` contains exactly one target and one advance condition (except the last one, where advance condition is optinal).

| Field | Type | Description |
|-------|------|-------------|
| `pressure` | Integer | Step target pressure in millibars. Mutually exclusive with `flowRate`. |
| `flowRate` | Integer | Step target flow rate in milligrams per second. Mutually exclusive with `pressure`. |
| `stepTime` | Duration | Advance after the given step-local duration in seconds. |
| `totalTime` | Duration | Advance when the total brew time reaches the given duration in seconds. |
| `totalWeight` | Integer | Advance when total output weight reaches the given value in milligrams. |

Duration values use the protobuf JSON string format, for example `"5s"`, `"30s"`, or `"4.5s"`.

---

## Example profile

The following example matches the profile shape accepted by `set-profile`:

```json
{
  "name": "Espresso",
  "temperature": 93000,
  "totalWeight": 36000,
  "steps": [
    {"pressure": 2000, "stepTime": "5s"},
    {"pressure": 9000, "totalWeight": 36000}
  ]
}
```

This profile starts with a low-pressure step for five seconds and then advances to a higher-pressure step until the shot reaches the target weight.

## Event stream

The `events` command subscribes to the daemon's event stream and runs until interrupted.

Each event carries exactly one changed state. The possible message payloads are:

| Field | Description |
|-------|-------------|
| `powerState` | Power on/off state change. |
| `brewProgress` | Brew progress update, if `brewStepIndex` isn't set, brew has been stopped. |
| `steamModeState` | Steam mode state change. |
| `userSettings` | Updated user settings. |
| `brewProfile` | Updated brew profile. |

---

## Metrics stream

The `metrics` command subscribes to the daemon's metrics stream and runs until interrupted.

Each metrics message includes:

| Field | Type | Description |
|-------|------|-------------|
| `temperature` | Integer | Current measured temperature in milli °C. |
| `pressure` | Integer | Current measured pressure in millibars. |
| `weight` | Integer | Current measured weight in milligrams. |
| `flowRate` | Integer | Current measured flow rate in milligrams per second. |
| `pidMetrics` | Optional object | PID controller internals when requested. |

When present, `pidMetrics` contains the following fields:

| Field | Description |
|-------|-------------|
| `p` | Proportional term. |
| `d` | Derivative term. |
| `i` | Integral term. |
| `f` | Feed-forward term. |
| `sum` | Combined controller output. |

## Metrics request options

The request generated by `openpresso-ctl metrics` contains two tunable fields.

| CLI option | Protocol field | Description |
|------------|----------------|-------------|
| `--rate` | `updateRate` | Metrics update interval as a protobuf duration. |
| `--pid-source` | `pidSource` | Selects which controller's PID state should be included. |

Valid values for `--pid-source` are:

| Value | Enum mapping |
|------|--------------|
| `none` | `PidSource_UNSPECIFIED` |
| `brew` | `BrewTemperatureController` |
| `steam-preheat` | `SteamPreheatTemperatureController` |
| `steam-active` | `SteamActiveTemperatureController` |

If PID monitoring isn't enabled in daemon config, error will be returned.

## Practical examples

Read the active profile:

```sh
openpresso-ctl get-profile
```

Upload a profile from a file:

```sh
openpresso-ctl set-profile espresso.json
```

Watch for state changes:

```sh
openpresso-ctl events
```

Sample live metrics twice per second:

```sh
openpresso-ctl metrics --rate 0.5
```

Include brew PID diagnostics:

```sh
openpresso-ctl metrics --pid-source brew
```
